Keep in mind that the story that is being told here is a inverse of Dante's Inferno.

A conversation of the Systems journey from Operator to Architect. We will cover techniques and tools that apply to each of the job title stages. In addition we will talk about how systems automation isnâ€™t confined and is becoming relevant to the application development life-cycle. While this talk is very Unix family focused the general topics should apply to any operating system. Management is welcome to attend as the talk may give them an idea of what their teams should be doing and how to support their team.

We will not be dwelling long on on any one technology stack as we will favor a strong overview of what the journey should look like. Executing commands from notes, from memory, in scripts, as a part of tools, and via push button self-service will be our major mile markers.

We are going to talk about roles not titles.
Titles are the public representation of who we are.
Roles are what we actually do.

In-Scope Roles:

  • Systems Operator/Hobbyist
  • Systems Administrator
  • Software Developer
  • Systems Engineer
  • Software Engineer
  • Systems Architect
  • Software Architect

Out-of-Scope Roles:

Valid but beyond the focus for today.
  • Analyst
  • Technical Writer

Systems Operator/Hobbyist:

  • Monitors the systems.
  • Little domain knowledge.
  • Executes prepared tasks. (recipes)
  • Tier 1 Support
  • Uses recipes to accomplish tasks.
  • Produces: buffering for senior co-worker
  • Produces: explores and learns new things
  • Run books and web searches.

Examples:

how to install postfix mail server ubuntu 13.10
how to start with arduino
iOS xcode hello world

HowtoForge

Systems Administrator:

  • Maintains the systems.
  • Knows where to start looking for a solution.
  • Has accumulated knowledge and builds from previous experiences.
  • Tier 2 Support
  • Fires over deploy and maintenance queues.
  • Produces: hopefully, stable working systems
  • Scripting to make tasks easier.

Systems Administrator: Automation through Scripting

Most artifacts are not maintained by the organization but live in home directories.

  • bash
  • Python
  • Ruby
  • Perl

Example:


#!/usr/bin/env bash

OPTIONS="-i ${HOME}/.vagrant.d/insecure_private_key"
OPTIONS="${OPTIONS} -o UserKnownHostsFile=/dev/null"
OPTIONS="${OPTIONS} -o StrictHostKeyChecking=no"

for i in $( seq 0 2 ); do
    ssh ${OPTIONS} vagrant@192.168.251.1${i} 'hostname' 2>/dev/null
done

Software Developer:

  • Maintains the applications.
  • Tier 2 Support.
  • Maintenance over features.
  • Produces: hopefully, stable software

Systems Engineer:

  • Builds the systems.
  • Understands the domain specific bigger picture.
  • Tier 3 Support.
  • Queues over fires.
  • Produces: documentation / recipes / inital running systems
  • Automation and Orchestration

Software Engineer:

  • Builds the applications.
  • Features over maintenance.
  • Tier 3 Support.
  • Produces: software to be implemented by Systems Engineers
  • Overlapping skill sets with Systems Engineers (scale/focus)

Systems/Software Architect:

  • Designs the Systems/Software.
  • Fully understands the bigger picture.
  • Produces: roadmaps / white papers / mock-ups / prototypes
